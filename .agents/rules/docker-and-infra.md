# Docker & Infrastructure

Triggered when editing Dockerfiles, compose configs, or environment variables. Defines the core environment separation and network isolation principles of the monorepo.

---

## 1. Environment & Container Management
- **Mock-First Local Development**: You MUST use the `mock` Docker Compose profile (`compose.sh mock`) in development. It replaces JVM/Jenkins and `agent-control` with a lightweight Python simulator. Always request permission from the user before launching the real Jenkins and agent containers.
- **GitHub Actions for agent-control Builds**: The `agent-control` service carries a heavy multi-gigabyte build context (Android and Flutter SDKs). Never build the `agent-control` image locally; delegate compilation to the remote GitHub Action build-images workflow.
- **Targeted Container Rebuilds**: When modifying code, rebuild/relaunch strictly the specific containers affected by your changes. Do not recycle the entire compose network unnecessarily.
- **Cross-Platform Compatibility**: Use bash explicitly to execute shell (`.sh`) scripts on Windows systems to maintain cross-platform build pipeline consistency.

---

## 2. Network & Volume Segregation
- **Host Exposure Isolation**: Only `gateway:80` (Caddy Ingress Gateway, mapped to `8880` locally) and standard `jenkins:8080` (for direct debugging) are exposed to the host system. Internal ports of Python microservices must remain closed.
- **Isolated Storage Volumes**: Each service strictly maintains its own dedicated, unshared Docker storage volume. Data crossing service boundaries must be transferred through secure HTTP control APIs (`/control/config`), never via direct multi-mount volumes on the host system.

---

## 3. Dynamic Environment Configuration
- **Centralized env file**: All runtime custom variables and secrets reside inside the unified `infra/compose.env` file.
- **Dynamic Examples**: The template `infra/compose.env.example` is auto-generated by parsing Python code schemas (`scripts/gen_env_examples.py`). Regenerate this file after making schema-altering updates.
- **Vite Cache-Busting**: Vite must generate content-hashed sub-resource assets for robust cache invalidation, while the HTML entrypoint remains `no-cache` to force immediate runtime revalidation.
